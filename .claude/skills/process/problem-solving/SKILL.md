# Problem Solving

## When To Use

Scope notes: Six structured techniques for breaking through stuck points — usable by planner, debugger, researcher, and implementer agents.

- Stuck for 3+ attempts on the same problem
- A solution feels forced or overly complex
- Complexity keeps growing with each iteration
- The team says "we've tried everything"

## Technique Dispatch

| Symptom | Technique | Reference |
|---------|-----------|-----------|
| Too complex, special cases multiplying | Simplification cascades | `references/simplification-cascades.md` |
| Can't see the answer, conventional approaches exhausted | Inversion | `references/inversion.md` |
| Feels similar to a solved problem elsewhere | Meta-pattern recognition | `references/meta-pattern.md` |
| Need a breakthrough, incremental won't work | Collision-zone thinking | `references/collision-zone.md` |
| Need to think at a different scale | Scale game | `references/scale-game.md` |
| Nothing is working, completely stuck | When-stuck dispatch | `references/when-stuck.md` |

## Protocol

1. Pick one technique based on the symptom table
2. Apply it for one focused attempt (time-box: don't spiral)
3. Evaluate the result — did it shift your understanding?
4. If no progress, switch to a different technique
5. After trying 3 techniques without progress, step back and question whether the problem statement itself is correct

## Rules

- One technique at a time — do not combine
- Write down what you learned from each attempt, even if it didn't solve the problem
- If a technique produces a breakthrough, document it in the plan's decisions log

## Gotchas

- Do not use problem-solving techniques as a substitute for reading the error message. Most issues have a direct cause — use techniques only when the direct path is exhausted.
- Techniques are time-boxed. Spending 30 minutes on collision-zone thinking without progress means switch technique, not push harder.
- Inversion can generate anxiety ("here are all the ways this fails") — treat the output as a solution roadmap, not a risk inventory.
- Meta-pattern recognition requires genuine cross-domain experience. If you can only find the pattern in one domain, it's not a meta-pattern — it's a specific solution.
