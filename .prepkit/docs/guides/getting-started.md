# Getting Started

PrepKit should get a new project to the right intent or delivery mode in a few minutes.

> **Note:** This is the generic, cross-host PrepKit installer reference (advanced / maintainers).
> The active PrepEdu kit ships only the `marketing` + `customer-prepedu` packs; some examples below
> describe the broader upstream PrepKit catalog. For day-to-day marketing use, see the
> [Team User Guide](./marketing-user-guide.md) and [Install & Setup](./installation.md).

## 1. Install

Run this from your project directory:

```bash
curl -fsSL https://raw.githubusercontent.com/namht1st/prep-kit/main/install.sh | bash
```

The installer auto-detects your GitHub token from the [gh CLI](https://cli.github.com) if installed. Otherwise it prompts for a [personal access token](https://github.com/settings/tokens) with `repo` scope. After setup, either open Claude Code and run `/quickstart`, or open the project in Codex and let it read `AGENTS.md`, discover repo skills from `.agents/skills/`, and use project-scoped subagents from `.codex/agents/`.

**Private / token-gated repos** — if you don't have `gh` installed, set the token manually:

```bash
export GITHUB_TOKEN=ghp_xxx
curl -fsSL -H "Authorization: token $GITHUB_TOKEN" \
  https://raw.githubusercontent.com/namht1st/prep-kit/main/install.sh | bash
```

Install to a specific directory:

```bash
curl -fsSL https://raw.githubusercontent.com/namht1st/prep-kit/main/install.sh | bash -s -- ./my-project
```

Pin a version or skip prompts:

```bash
curl -fsSL ... | bash -s -- --version v1.5.0 --type engineering --force .
```

### Alternative: standalone binary

Install PrepKit once, then use `prepkit` anywhere:

```bash
curl -fsSL https://raw.githubusercontent.com/namht1st/prep-kit/main/install-standalone.sh | bash
prepkit new my-project
# or:
mkdir my-project && cd my-project && prepkit init --preset solo-engineer
```

When a new PrepKit release is available, upgrade the standalone install and refresh the project:

```bash
prepkit upgrade
cd your-project
prepkit init
```

For a genuinely greenfield app or service, prefer the guided path first:

```bash
prepkit setup
```

The guided flow can recommend a stack from your project goal, record that decision in `.prepkit/kit-state.json`, and tell you to bootstrap the real app before you layer PrepKit on top with `prepkit init`.

Available presets:
- `solo-engineer`: engineering + frontend + databases + system-design packs, `patch` default mode
- `product-team`: engineering + frontend + product + databases + system-design packs, `build` default mode
- `full-stack`: engineering + frontend + product + marketing + databases + system-design packs, `build` default mode
- `prepedu-ai-product`: AI/ML + product + engineering + Node/Python backend + databases + PrepEdu customer context, `build` default mode

When using npm init, run `/quickstart` in Claude Code after init to complete setup. If you use Codex instead, open the repo there and it will read `AGENTS.md` automatically, with repo skills and project subagents available after the initial build. Both hosts share the same plan files, state files, and generated pack selection.

### Solo vs team mode

In **solo** mode, PrepKit runtime surfaces (`.prepkit/`, `.claude/`, `.codex/`, `.agents/`, `plans/`, `CLAUDE.md`, `AGENTS.md`) are added to `.gitignore` so they stay local and don't appear in your project's git history. In **team** mode, these are tracked so teammates share the same setup. To switch from solo to team later, remove the PrepKit lines from `.gitignore` and commit the files.

### What gets created

```
your-project/
├── AGENTS.md          ← Portable agent instructions generated by PrepKit
├── .agents/
│   └── skills/        ← Repo-scoped Codex skills (generated symlinks)
├── .claude/            ← Claude Code config, hooks, commands, skills
│   ├── commands/       ← Slash commands (/plan, /review, /status, etc.)
│   ├── hooks/          ← Session lifecycle hooks
│   ├── skills/         ← Process and domain skills
│   └── settings.json   ← Generated settings (do not edit directly)
├── .codex/
│   └── agents/        ← Project-scoped Codex subagents generated from PrepKit roles
├── .prepkit/           ← Kit source, generated indexes, and runtime state
│   ├── active.manifest.json
│   ├── docs/
│   ├── kit-state.json
│   ├── pack-selection.json
│   ├── packs/
│   ├── presets/
│   └── scripts/
├── plans/              ← Work tracking
│   ├── active/         ← Current initiatives
│   ├── archive/        ← Completed work
│   └── research/       ← Discovery artifacts
└── .prepkit/kit.manifest.json   ← Kit source of truth
```

## 2. Recommended Claude Code Path

Claude Code is the slash-command and hook host:

1. Run `/prep-doctor` on first use to verify the kit's health, and `prepkit next-step` to inspect the current plan state.
2. Run `prepkit doctor` from the terminal to see selected pack command groups; activate `engineering`, `product`, `backend`, or `databases` packs only when you need those slash workflows.
3. Route product and technical work through facilitation skills first; keep decisions in plan/spec/knowledge files rather than chat history.
4. Use `/prep-plan` and `prepkit close` for the main delivery loop.

## 3. Recommended Codex Path

PrepKit keeps Codex on the same file-backed operating model as Claude Code. The recommended path is:

1. Let Codex read `AGENTS.md`, then skim `.prepkit/docs/reference/codex-catalog.md` for the current skill and role catalog.
2. Check `.prepkit/kit-state.json`, `.prepkit/pack-selection.json`, `plans/active/`, and run `node .prepkit/scripts/prepkit-cli.mjs next-step` before broad changes. If multiple active plans exist, bind one with `node .prepkit/scripts/prepkit-cli.mjs bind <plan>`.
3. Create tracked work with `node .prepkit/scripts/prepkit-cli.mjs plan "<task>"`. If the work is ambiguous or spec-first, refresh design artifacts with `node .prepkit/scripts/prepkit-cli.mjs init-spec --plan <plan>`.
4. Use `$prepkit-navigator` when the route is unclear, use repo skills from `.agents/skills/` for targeted help, and use `.codex/agents/` when you want explicit PrepKit roles.
5. Rebuild with `node .prepkit/scripts/prepkit-cli.mjs build`, validate with `node .prepkit/scripts/prepkit-cli.mjs validate`, and close with `node .prepkit/scripts/prepkit-cli.mjs close`.

## 4. Route The Work First

PrepKit splits work into separate intents and delivery modes.

In Codex, use the same plan files through the repo CLI entrypoint (`prepkit plan`, `prepkit next-step`, `prepkit close`) instead of assuming Claude-only slash commands are available. Codex reads `AGENTS.md` and can use explicit subagents from `.codex/agents/`.

Separate intents do not start delivery:

- `review` via `/review`
- `explain` and `research` as navigator-routed intents, not standalone core slash commands

Delivery work stays explicit:

- `patch`: one or two low-risk files with a clear path
- `build`: standard implementation flow; default here once the change is larger than a quick fix
- `design`: spec-first work for ambiguous or cross-cutting changes

For most implementation work, start with:

```bash
node .prepkit/scripts/prepkit-cli.mjs plan "<task>"
```

For work that should start with proposal and design artifacts, use:

```bash
node .prepkit/scripts/prepkit-cli.mjs plan --mode design "<task>"
node .prepkit/scripts/prepkit-cli.mjs init-spec --plan <plan>
```

If the route is unclear, let navigator-driven planning sort it out instead of guessing.

## 4. Deliver The First Change

Typical flow:

1. Run `/prep-plan <task>` or `prepkit plan "<task>"`.
2. Follow the generated `plan.md` and `spec/` artifacts.
3. Work through approved scope step by step.
4. Use `prepkit next-step` to surface the next move as the work progresses.
5. Use `prepkit close` when the work is done and validated.

## Model Profiles

PrepKit defines three model profiles in `.prepkit/kit.manifest.json` under `modelProfiles`: `quality`, `balanced`, and `budget`. The `defaultModelProfile` key selects the active profile.

To switch profiles, edit `defaultModelProfile` in `.prepkit/kit.manifest.json` and rebuild:

```bash
# Example: switch to the quality profile
# Edit .prepkit/kit.manifest.json -> "defaultModelProfile": "quality"
node .prepkit/scripts/prepkit-cli.mjs build
```

The build resolves each agent's model from the active profile and writes it into the generated agent files.

PrepKit also supports optional task-aware routing on top of the active profile through `modelRouting`:

```json
{
  "modelRouting": {
    "modeOverrides": {
      "design": {
        "planner": "opus",
        "researcher": "opus"
      }
    },
    "laneOverrides": {}
  }
}
```

Resolution order is:

1. explicit `agents[].model`
2. matching `modeOverrides`
3. matching `laneOverrides`
4. active profile assignment
5. built-in fallback

Mode-aware overrides only apply when `PREP_PLAN_MODE` is set during the build. If no explicit build mode is provided, the build falls back to lane and profile resolution.

The shipped core manifest keeps profile switching meaningful by default. Add lane overrides only when you want a specific lane to ignore the selected profile on purpose.

## Rebuilds And Recovery

PrepKit generates runtime files such as:
- `.claude/.prep.json`
- `.claude/settings.json`
- `.claude/capabilities.json`
- `.prepkit/active.manifest.json`

If those files go missing, session startup rebuilds them automatically. Manual rebuild commands still work:

```bash
node .prepkit/scripts/prepkit-cli.mjs build
node .prepkit/scripts/build-pack.mjs
node .prepkit/scripts/prepkit-cli.mjs validate
```

## Troubleshooting

### `prepkit: command not found` after standalone install

The installer links the CLI into `~/.local/bin` and now tries to register that directory in your shell config automatically. Your current shell still needs to reload the config before `prepkit` is available.

**Fix:** reload your shell config:

```bash
source ~/.zshrc
# or
source ~/.bashrc
```

Or use the full path: `~/.local/bin/prepkit init`

You can also create a fresh workspace directly with `prepkit new my-project`.

### Build fails during init

If Node.js is not installed (standalone mode), the build is skipped with a warning. Install Node.js from [nodejs.org](https://nodejs.org), then run the build manually:

```bash
node .prepkit/scripts/prepkit-cli.mjs build
```

If using a preset:

```bash
node .prepkit/scripts/build-pack.mjs --preset <preset-name>
# or for individual packs:
node .prepkit/scripts/build-pack.mjs --packs engineering
```

### `Could not locate PrepKit source`

This means the CLI cannot find `.prepkit/kit.manifest.json`. Common causes:
- Running `prepkit` from outside a PrepKit directory without `PREPKIT_HOME` set
- Corrupted standalone install

**Fix:** Set `PREPKIT_HOME` to your install root, or re-run the installer.

### Verify your setup

In Claude Code, run `/doctor` to check kit health. From the terminal:

```bash
node .prepkit/scripts/prepkit-cli.mjs doctor
node .prepkit/scripts/prepkit-cli.mjs validate
```
